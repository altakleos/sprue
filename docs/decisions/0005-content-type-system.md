# ADR-0005: Content Type System — Types, Schemas, and Size Profiles

## Context
The KB started with 5 page types and hardcoded word-count limits. As content diversified, the fixed types couldn't express the difference between a concept explainer and a tool reference, and uniform size limits forced long topics into artificially short pages. Types were also labeled "epistemic categories," which confused users about whether the label described the content's form or its knowledge status.

## Decision
Expand to 9 content types, reframed as content types (not epistemic categories) to clarify they describe form, not certainty. Extract all type definitions to types.yaml as the single source of truth, with per-type size profiles (compact, standard, extended) and section contracts. Add a provenance field (sourced vs. synthesized) to track whether content originated from external sources or was generated by the agent.

## Alternatives Considered
- **Keep 5 types with flexible word counts** — rejected because it overloaded existing types with incompatible section expectations
- **Freeform types defined per-page** — rejected because it would eliminate the consistency that section contracts provide

## Consequences
The type system became the structural backbone for compilation — the agent knows exactly what sections to produce and how long each type should be. The trade-off is that adding a new type requires defining its full schema (sections, size profile, description), which raises the bar for type creation.
