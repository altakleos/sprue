# Verify Protocol

*Requires `AGENTS.md` and `sprue/engine.md` in context (loaded via bootstrap).*

**Trigger:** "verify", "verify --page kafka", "verify 5", "verify --tier critical"

## Role

Independently verify wiki page claims against source material. VERIFY reads the
wiki page and its raw source, uses LLM judgment to extract and assess claims,
proposes fixes for stale or wrong content, and updates verification state.

**Key difference from maintain:** Maintain is janitorial (lint, broken links, frontmatter).
Verify is epistemic — it checks whether the *content* is true.

---

## Prerequisites

Before every run:
1. Read `memory/rules.yaml` (structural rules)
2. Read `memory/corrections.md` (active factual corrections)

---

## Phase 1: Prioritize

Select which pages to verify.

1. Run `python3 sprue/scripts/prioritize.py --top 20 --json` to get scored candidates
2. If `--page <slug>` was specified, skip scoring — verify that page directly
3. If a number was given (`verify 5`), take the top 5 by score
4. If `--tier <tier>` was specified, pass `--tier` to the script

Present the selection:

```
🔍 Verification candidates (scored from N pages, M unverified):

 #  Page                    Risk        Score  Signal
 1  supply-chain-security   critical    82.5   never verified, fast decay
 2  cicd                    critical    81.8   never verified, 35 inbound links
 3  iam                     critical    77.2   never verified, 23 inbound links
```

**Manual mode (default):** STOP. Wait for approval. Accept: `all`, `1,3,5`, `1-5`, `none`.
**Semi mode (--semi):** LLM picks top N from `instance/config.yaml` → `verify.semi.max_pages`. Show selection. No stop.
**Auto mode (--auto):** LLM picks top N from `verify.auto.max_pages` above `verify.auto.min_score`. No stop. Post-hoc report at end.
**Adversarial mode (--adversarial):** Orthogonal to the above — adds a writer/critic/judge loop to Phase 3 claim assessment. See §Adversarial Mode below. Composes with manual/semi/auto.

---

## Phase 2: Gather Baseline Sources

For each selected page, gather initial source material. This is not the final
source set — Phase 3 will fetch additional targeted sources per-claim.

### Step 2a: Raw source file

1. Look up this page's slug in `wiki/.index/by-slug-raws.yaml` (derived index, generated by `sprue/scripts/build-index.py`)
2. Read the raw file(s) listed under the slug
3. If multiple raw files map to one wiki page, read all of them
4. If the slug is absent from the index, the page has no raw source — skip Tier 1 and proceed to Tier 2 (see Edge Cases)

The raw file is context — the original article the page was compiled from. It may
be thin (e.g., a Wikipedia overview) and insufficient for verifying specific claims.
That is expected. Don't stop here.

> The index is derived from `instance/state/compilations.yaml` filtered against the
> current manifest, so orphaned compilation rows (from deleted or merged pages) don't
> appear here. Do not scan `compilations.yaml` directly for this lookup.

### Step 2b: Original source URL

1. Search `instance/state/imports.yaml` for the entry matching the raw file path
2. Note the `source` URL — this is available for re-fetching if needed in Phase 3

### Step 2c: Authoritative documentation registry

1. Read the page's `topic` values from the manifest
2. Look up each topic in `instance/sources.yaml`
3. Note which authoritative URLs are available — fetch them in Phase 3 as needed

Report what's available (not what's been fetched yet):

```
📚 Sources available for supply-chain-security:
   Raw: raw/articles/supply-chain-security-2026-04-12-abc123.md (50 lines)
   URL: https://en.wikipedia.org/wiki/Software_supply_chain
   Authoritative: no sources.yaml entries for [sbom, slsa, sigstore]
```

## Phase 3: LLM Verify (Extract → Source → Assess loop)

For each page, extract claims first, then find sources for each claim, then assess.
This is a **claim-driven loop** — every factual claim must be sourced before it can
be assessed. Do not batch-assess all claims against a single source.

### Step 3a: Read the page

Read the full wiki page content.

### Step 3b: Read raw source

Read the raw file(s) from Phase 2a.

### Step 3c: Extract verifiable claims

Identify claims that are potentially checkable:

**Extract these:**
- Version numbers ("Kafka 3.6", "TLS 1.3", "since Kubernetes 1.24")
- Default values ("retention defaults to 7 days", "auto-commit is enabled by default")
- Configuration settings ("set MaxAuthTries 3", "replication.factor=3")
- Performance numbers ("p99 < 100ms", "drops throughput 20-30%", "max 120 partitions")
- Behavioral assertions ("auto-commit commits before processing completes")
- Security properties ("RSA key exchange has no forward secrecy")
- Limit/capacity claims ("max message size 1MB", "up to 10,000 connections")
- Protocol specifics ("TLS 1.3 handshake is 1-RTT", "uses Raft consensus")
- Named events with dates ("SolarWinds attack 2020", "GhostAction attack 2025")
- Architectural recommendations backed by specific claims ("use MSK — self-managed Kafka is a full-time job")
- Code examples with specific API parameters, CLI flags, or config keys
- References to standards or specifications ("ISO/IEC 5962:2021", "RFC 8446")

**Skip these:**
- Pure opinions without factual backing ("Kafka is the best streaming platform")
- Tautological definitions ("a load balancer distributes load")
- General principles too broad to verify ("security is important")
- Subjective emphasis ("certificate expiry is THE #1 cause")
- Hedged experience-based estimates ("typically cuts 2-5 minutes", "catches ~90%")

### Step 3d: Source each claim (escalation ladder)

For EACH extracted claim, attempt to find an authoritative source. Exhaust every
tier before marking a claim unverifiable. **Do not skip tiers.**

```
For each claim:
  1. Check raw file → does it address this specific claim?
     YES → use as source, assess claim
     NO  → continue to next tier

  2. Check instance/sources.yaml URLs → fetch via Jina Reader, does it address this claim?
     YES → use as source, assess claim
     NO  → continue to next tier

  3. Web search → construct a targeted query for this specific claim:
     - "[technology] [specific feature/setting] official documentation"
     - "[technology] [version/default/limit] docs"
     - "[named event] [year] CVE details"
     - "[standard name] [standard number]"
     Fetch the top result from an authoritative domain (.org, .io, .dev, docs.*)
     via Jina Reader or WebFetch.
     FOUND → use as source, assess claim
     NOT FOUND → continue to next tier

  4. Training knowledge (last resort) — only if tiers 1-3 all failed for this claim.
     - Mark the claim assessment with `source: training_knowledge`
     - If the claim is a hard factual assertion (version, default, limit), it stays
       `unverifiable` — training knowledge alone cannot confirm factual claims
     - If the claim is a behavioral or architectural assertion that the LLM is
       confident about, it may be assessed but the source must be noted
```

**Batching for efficiency:** Group claims by topic/technology when constructing web
searches. One well-chosen authoritative page often covers multiple claims. For example,
the AWS IAM quotas page covers all policy size limits in one fetch.

**Authoritative domain preference for web searches:**
- Official project sites (.org, .io, .dev) — highest authority
- Official vendor docs (docs.aws.amazon.com, kubernetes.io) — highest authority
- GitHub repos (README, docs/) — high authority for OSS projects
- Wikipedia — good for dates, events, standards, organizations
- Reputable security blogs (wiz.io, paloaltonetworks.com) — good for CVEs, incidents
- Avoid: random blogs, StackOverflow answers, marketing pages

### Step 3e: Assess each claim

Single-pass assessment (the default) uses the **writer** role only — read `sprue/prompts/verify-writer.md` and apply it per claim. Adversarial mode augments this with critic and (conditionally) judge; see §Adversarial Mode below.

After sourcing, assign a verdict:

| Verdict | Meaning | Action |
|---|---|---|
| **confirmed** | Authoritative source explicitly supports this claim | No change |
| **stale** | Claim references a version/value that has changed | Fix with current value |
| **wrong** | Authoritative source directly contradicts this claim | Fix with correct value |
| **unverifiable** | All 4 source tiers exhausted, no source addresses this claim | See below |

**Handling unverifiable claims:**
- If the claim is hedged experience-based guidance ("typically", "~90%", "2-5 minutes"),
  accept it as opinion — it doesn't block `confidence: high`
- If the claim is a hard factual assertion (specific number, specific event, specific
  behavior) that no source could confirm, it blocks `confidence: high` for the page
- In either case, note the claim in the verification report

**Rules:**
- Only mark `wrong` if you have a specific source excerpt that contradicts the claim
- When in doubt after exhausting sources, mark `unverifiable` — never guess
- Temporal claims ("since version X", "as of 2024") are `stale` unless confirmed
- Never flag opinions or architectural recommendations as "wrong"
- Hedged claims ("often", "typically") get more latitude than absolute claims

### Step 3f: Propose fixes

For each `stale` or `wrong` claim:

1. **Quote the source excerpt** that contradicts it
2. **Cite the source URL** or raw file path + line number
3. **Propose the surgical fix** — specific sentence or value, not a section rewrite
4. **Classify the error type:**
   - `stale_version` — version number outdated
   - `wrong_default` — default value incorrect
   - `wrong_limit` — capacity/limit claim wrong
   - `wrong_behavior` — behavioral assertion incorrect
   - `wrong_name` — incorrect name for an event, tool, or concept
   - `stale_recommendation` — guidance based on old context
   - `security_misstatement` — security property wrong (highest severity)

**CRITICAL: Never fix without source evidence.** If you can't point to a specific
excerpt in a fetched source, the claim stays `unverifiable`.

### Confidence upgrade criteria

**This is the sole path to `confidence: high` for `author: llm` pages.** Compile does not promote (see `sprue/engine.md` Confidence Invariant). Every high-confidence LLM page was promoted here after source-backed fact-checking.

A page qualifies for `confidence: high` when:
- All hard factual claims are either `confirmed` or `fixed` with authoritative sources
- Remaining `unverifiable` claims are only hedged experience-based guidance (not hard facts)
- No `security_misstatement` errors were found (or all were fixed)

A page stays at `confidence: medium` when:
- One or more hard factual claims remain unverifiable after exhausting all source tiers
- Fixes were applied using only training knowledge

A page drops to `confidence: low` when:
- Multiple claims were found `wrong` without clear fixes
- Major structural issues or contradictions between sections

## Phase 4: Apply + Log

### Auto-fix (no approval needed)

- Version numbers where source clearly shows a different current version
- Default values where source documents a different default
- Deprecated flags/APIs where source marks them deprecated
- Add inline source citation after each fix

### Needs approval

- Removing or significantly changing security assertions
- Changing architectural recommendations
- Any fix where the source is ambiguous
- Fixes where raw file and authoritative docs disagree
- Fixes sourced only from LLM training knowledge (Tier 4)

Present findings:

```
📝 Verification: supply-chain-security

Claims checked: 8
  ✅ Confirmed: 5
  🔧 Auto-fixed: 2
  ⚠️ Needs approval: 1
  ❓ Unverifiable: 0

Auto-applied:
  Line 34: "Sigstore 1.0" → "Sigstore 1.2" (stale_version)
    Source: raw/articles/supply-chain-security-2026-04-12.md line 89

Needs approval:
  1. Line 67: Claims "SLSA Level 3 requires hermetic builds" — source says
     Level 3 requires "isolated" builds, not necessarily hermetic. Change?

Accept fixes? [all / 1 / none]
```

### Update frontmatter

After all fixes applied:

- Set `last_verified: YYYY-MM-DD` (today's date)
- Adjust `confidence` per the criteria in Step 3 → Confidence upgrade criteria

### Update state ledger

Append to `instance/state/verifications.yaml`:

```yaml
- verified_at: '<ISO8601>'
  mode: manual | semi | auto      # add `+adversarial` suffix when adversarial mode was composed (e.g. `auto+adversarial`)
  page: <slug>
  wiki_slug: <slug>
  risk_tier: <tier>
  source_tiers_used: [raw, authoritative]
  sources:
    - type: raw
      path: <raw file path>
  claims_checked: N
  confirmed: N
  fixed: N
  unverifiable: N
  confidence_before: <old>
  confidence_after: <new>
  fixes:
    - section: "## Section"
      old: "wrong claim"
      new: "correct claim"
      error_type: <type>
  priority_score: N.N
  # Adversarial-mode fields (omit when not used):
  claims:                                   # per-claim audit trail
    - claim: "<verbatim claim text>"
      source_tier_used: raw | sources.yaml | web | training
      writer_verdict: confirmed | stale | wrong | unverifiable
      critic_invoked: true | false
      critic_verdict: <same set> | cannot_rebut | needs_more_sources | null
      judge_invoked: true | false
      judge_verdict: <same set> | null
      final_verdict: <the operative one>
  promotion_blocked_by_critic: N            # count of claims where judge ruled against writer and that prevented/downgraded a promotion
```

The `claims:` block and `promotion_blocked_by_critic` counter are populated only when adversarial mode was used. Older entries without these fields parse cleanly — the schema is additive.

### Update memory log

Append to `memory/log.jsonl`:

```json
{"ts":"<ISO8601>","op":"verify","title":"<slug>","created":0,"modified":N,"deleted":0,"summary":"Verified N claims. Fixed M. Confidence: old → new."}
```

### Update corrections

If a fix corrects a systematic LLM error (the LLM would likely reproduce this mistake),
append to `memory/corrections.md` following its existing format.

### Respect existing corrections

Before any write, check `memory/corrections.md`. Active corrections for the target
page take precedence over any proposed fix.

## Batch Mode

`verify 5` or `verify --semi 10` or `verify --auto 20`:

- Phase 1 runs once (score all pages, select N)
- Phases 2-4 run per page, sequentially
- State writes happen per page (crash-safe — if interrupted, completed pages are recorded)
- Cumulative summary at the end:

```
✅ Verification batch complete: 5 pages

 Page                    Claims  Fixed  Unverifiable  Confidence
 supply-chain-security   8       2      0             medium → high
 cicd                    11      0      0             medium → medium
 iam                     6       1      0             medium → high
 kafka                   19      3      2             medium → medium
 kubernetes              16      1      0             medium → high

Total: 60 claims checked, 7 fixes, 2 unverifiable
```

## Modes

| Mode | Invocation | Candidate gate | Fix gate |
|---|---|---|---|
| **manual** (default) | `verify` | User picks from list | User approves non-auto fixes |
| **semi** | `verify --semi` | LLM picks top N | User approves non-auto fixes |
| **auto** | `verify --auto` | LLM picks top N | Auto-fix all with source evidence |
| **adversarial** | `verify --adversarial` (composable) | Inherits from paired mode | Inherits; claim verdicts come from writer/critic/judge loop |

Semi and auto read `instance/config.yaml` → `verify.semi.max_pages` / `verify.auto.max_pages`.

Auto mode produces a post-hoc report (no stops) but still respects:
- `verify.auto.min_score` — skip pages below this score
- Never auto-fix without source evidence
- Never auto-fix security_misstatement errors (always flag)

## Adversarial Mode

The default verify loop is single-pass: one LLM reads the claim and sources and emits a verdict. That collapses close-but-not-matching source language, source silence, and implicit writer assumptions into false `confirmed` verdicts. Adversarial mode adds two more roles — **critic** and **judge** — to force those cases into the open.

### Roles (one prompt file each, all under `sprue/prompts/`)

| Role | Prompt | When invoked | Output |
|---|---|---|---|
| **writer** | `sprue/prompts/verify-writer.md` | Every claim (always) | verdict ∈ {confirmed, stale, wrong, unverifiable}, source_ref, rationale, proposed_fix |
| **critic** | `sprue/prompts/verify-critic.md` | Gated (see below) | rebuttal verdict or `cannot_rebut`; targets 5 named failure modes |
| **judge** | `sprue/prompts/verify-judge.md` | Only when critic returns a non-trivial rebuttal (NOT `cannot_rebut` and NOT `needs_more_sources`) | `final_verdict` (may match writer, critic, or a third option) |

### Gating — when to invoke critic and judge

Critic is expensive (~1× extra LLM pass per claim). Gate it:

| Trigger | Invoke critic? |
|---|---|
| `verify --adversarial` (explicit flag) | Always, for every claim |
| Page with `risk_tier: critical` | Yes, for every claim |
| Page with `risk_tier: operational`, claim has `error_type` candidate in {version, default, limit, behavior, security_misstatement} | Yes |
| Writer verdict = `unverifiable` | Always — cheapest high-value trigger (writer is least confident here; critic de-biases) |
| All other (reference tier, conceptual claims, writer `confirmed` on low-risk pages) | No (single-pass behavior) |

Judge is invoked **only when the critic returns a non-trivial rebuttal** (i.e., `critic_verdict` is a concrete verdict, not `cannot_rebut` or `needs_more_sources`). When the critic concedes, the writer's verdict is final.

### Verdict flow

```
writer → verdict_W

if gating_applies:
    critic sees writer's output → verdict_C
    if verdict_C in {cannot_rebut, needs_more_sources}:
        final = verdict_W   (writer stands)
    else:
        judge sees both → verdict_J
        final = verdict_J
else:
    final = verdict_W   (single-pass)
```

`final` is the operative verdict used to classify the claim, propose fixes, and drive the page's confidence transition in Phase 4.

### Calibration signal

Track `promotion_blocked_by_critic` per page in `instance/state/verifications.yaml` — the count of claims where the judge ruled against the writer in a way that prevented or downgraded a promotion. If this counter runs high across many pages, the writer is rubber-stamping and default gating should widen (e.g., turn on critic for operational-tier by default even without `--adversarial`).

### Composes with other modes

- `verify --semi --adversarial` — LLM picks candidates from top N; each claim gets adversarial review.
- `verify --auto --adversarial` — auto candidate selection + auto-fix with source evidence + adversarial claim review.
- `verify --adversarial --page kafka` — target one page, full adversarial.

## Edge Cases

### Page with no raw source
The slug is absent from `wiki/.index/by-slug-raws.yaml` (either human-authored or the page's compilation row was dropped as orphaned after a deletion/merge). Skip Tier 1. Try Tier 2 (re-fetch URL from imports.yaml — only available for compiled pages) → Tier 3 (sources.yaml) → Tier 4 (training knowledge). Cap confidence at `medium` if no source material was available.

### Page with multiple raw sources
Read ALL raw files. Claims may be verified against different sources — note which source per claim in the verification entry.

### Stale source URL (Tier 2 fetch fails)
Log the failure. Fall back to raw file (immutable, always available). If raw file is also missing, use Tier 3/4. Note `url_stale: true` in the entry.

### Recently verified page
Skip pages verified within `verify.cooldown_days` (default 30). Override with explicit `--page slug`.

### Contradictory sources
When raw file and authoritative docs disagree, prefer the authoritative doc (the raw file may have been wrong when imported). Log the contradiction.

---

## Constraints

- Never modify `raw/`, `notebook/`, or `inbox/`.
- Never create new wiki pages. Flag missing pages for import + compile.
- Maximum pages per auto run: `verify.auto.max_pages` from instance/config.yaml.
- Read `memory/rules.yaml` and `memory/corrections.md` before writing.
- Log every change to `memory/log.jsonl`.
- Append to `instance/state/verifications.yaml` AFTER each page's fixes are applied (crash-safe).
